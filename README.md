# Media Scanner Daemon

Высокопроизводительная системная служба для Linux, предназначенная для рекурсивного сканирования директорий на наличие медиафайлов (аудио, видео, изображения) и предоставления результатов через встроенный HTTP-сервер в формате JSON.

Архитектура сервера построена на базе epoll и неблокирующих сокетов, что позволяет выдерживать 10 000+. Журналирование интегрировано напрямую с syslog

---

## Требования

* ОС: Linux (из-за использования `epoll` и `systemd`)
* Компилятор с поддержкой **C++17** (GCC 8+ / Clang 7+)
* **CMake** 3.14 или выше
* Права `root` (sudo) для установки системной службы

---

## Сборка проекта

Проект использует CMake для управления зависимостями (nlohmann/json подтягивается автоматически).

1. Клонируйте репозиторий и перейдите в его директорию:
```bash
sudo git clone <URL_репозитория>
cd Media_Scanner

```


2. Сгенерируйте файлы сборки и скомпилируйте релизную версию:
```bash
sudo cmake -B build
sudo cmake --build build --config Release

```

После успешной сборки исполняемый файл `media_scanner` появится в директории `build/`.

---

## Конфигурация

Служба настраивается через конфигурационный файл.

1. Скопируйте файл конфигурации `include/media_scanner.conf` и переместите туда, откуда служба будет подтягивать файл. Например: `/etc/media_scanner.conf`

```bash
sudo cp include/media_scanner.conf /etc/media_scanner.conf

```

2. Отредактируйте следующие параметры (обязательно используйте **абсолютный путь** для директории сканирования):
```ini
# Абсолютный путь к директории с медиафайлами
path = /var/media_storage

# Интервал пересканирования директории (в секундах)
interval = 60

# Включить HTTP-сервер (1 - включен, 0 - выключен)
http_mode = 1

# Порт для входящих HTTP-подключений
port = 8080

```

## Установка и запуск (Systemd)

Чтобы программа работала непрерывно в фоне и автоматически запускалась при старте системы, ее необходимо зарегистрировать как службу `systemd`.

**Шаг 1: Копирование бинарного файла**
Перенесите собранный файл в стандартную директорию для пользовательских бинарников:

```bash
sudo cp build/media_scanner /usr/local/bin/

```

**Шаг 2: Создание Systemd-юнита**
Скопируйте файл службы:

```bash
sudo cp include/media_scanner.service /etc/systemd/system/media_scanner.service
```

Отредактируйте следующие параметры:

```ini
[Unit]
Description=Media Scanner HTTP Service (epoll)
After=network.target

[Service]
Type=simple
User=root
ExecStart=/usr/local/bin/media_scanner --config /etc/media_scanner.conf
Restart=on-failure
RestartSec=5
# Увеличиваем лимит файловых дескрипторов для поддержки 10000+ сессий
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target

```

**Шаг 3: Запуск службы**
Перезагрузите конфигурацию `systemd`, активируйте и запустите службу:

```bash
sudo systemctl daemon-reload
sudo systemctl enable media_scanner
sudo systemctl start media_scanner

```

Проверьте, что служба успешно запустилась:

```bash
sudo systemctl status media_scanner

```

## Мониторинг и Логирование

Служба не выводит данные в консоль, а отправляет их напрямую в системный журнал Linux через `syslog`.

Чтобы посмотреть логи в реальном времени, используйте утилиту `journalctl`:

```bash
sudo journalctl -u media_scanner -f

```

## Использование API

Когда служба запущена и HTTP-режим включен, вы можете получить список найденных медиафайлов с помощью простого GET-запроса.

Пример через `curl`:

```bash
curl http://localhost:8080/media_files

```

**Ожидаемый ответ (JSON):**

```json
{
  "audio": [
    "music/track1.mp3",
    "podcasts/episode2.wav"
  ],
  "images": [
    "vacation/photo.jpg",
    "design/logo.png"
  ],
  "video": [
    "movies/film.mp4"
  ]
}

```

*Примечание: Все пути в JSON-ответе являются относительными (отталкиваясь от базовой папки, указанной в конфиге).*


## Тестирование производительности и устойчивости

Для верификации способности демона стабильно удерживать более 10 000 одновременных соединений и проверки эффективности архитектуры на базе `epoll` было проведено комплексное асинхронное нагрузочное тестирование с помощью скриптов на Python (`asyncio`).

### 1. Стресс-тест: Удержание 10 000 активных сессий (Keep-Alive)
Цель эксперимента — симуляция непрерывного конкурентного взаимодействия реальных пользователей с сервером. Тестовый скрипт `load_tester1.py` устанавливал соединение пачками (до 1500 сокетов в секунду), после чего удерживал ровную «полку» в 10 000 параллельных сессий в течение 20 секунд. Внутри каждой сессии отправлялись циклические HTTP-запросы `GET /media_files` с заголовком `Connection: keep-alive` и фиксированной паузой в 50 мс между запросами.

Каждый из 10 000 пользователей начинает непрерывный цикл обмена сообщениями с сервером:
1. Формируется полноценный HTTP-запрос `GET /media_files` с заголовком `Connection: keep-alive`.
2. Запрос отправляется в сокет; запускается высокоточный таймер.
3. Сервер через `epoll_wait` обрабатывает готовые к чтению сокеты, моментально вычитывает заголовки, извлекает JSON-строку из `MediaState` по
4. Скрипт считывает ответ до завершающих символов заголовка `\r\n\r\n`. При успешном статусе `200 OK` фиксируется время задержки (Latency).

**Итоговые показатели теста производительности (`load_tester1.py`):**
* **Успешных активных сессий:** 10 000 (0% потерь соединений).
* **Всего обработано HTTP-запросов за время теста:** 172 077.
* **Средняя интенсивность обработки (RPS):** 8 603 запроса в секунду.
* **Среднее время ответа сервера (Latency):** 543.23 мс.


### 2. Тест деградации: Зависимость задержки от нагрузки (Latency vs Load)
Цель эксперимента — определить «точку перелома» производительности системы. Скрипт `load_tester2.py` пошагово увеличивал количество одновременно активных пользователей от 2 000 до 15 000 с ограничением скорости открытия сокетов до 1500 в секунду.

После получения ответа соединение **не разрывается**. Виртуальный пользователь «засыпает» ровно на **50 миллисекунд** (`asyncio.sleep(0.05)`). Это критически важный шаг методики, позволяющий симулировать реальную интерактивную работу клиентов и проверять, как `epoll` справляется со смесью «активных» и «спящих» дескрипторов.

**Результаты замера среднего времени ответа при ступенчатом росте нагрузки:**
* **2 000 одновременных пользователей:** 144.41 мс.
* **4 000 одновременных пользователей:** 183.85 мс.
* **6 000 одновременных пользователей:** 516.94 мс.
* **8 000 одновременных пользователей:** 456.09 мс.
* **10 000 одновременных пользователей:** 1282.61 мс.
* **15 000 одновременных пользователей:** 1479.47 мс.
